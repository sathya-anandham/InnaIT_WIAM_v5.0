A. EXISTING PLATFORM CONTEXT
======================================================================

The platform already supports:

- Account management

- Multi-tenant Oracle architecture

Goal:
Add an enterprise-grade Device Registry + Device Assignment capability so that devices are first procured/registered by the enterprise, then mapped to a user/account before the device is delivered or activated.

In addition, enable Magic Link as the first-time bootstrap login method for users who have an assigned FIDO device but have not yet enrolled that device.

======================================================================
B. REQUIRED BUSINESS FLOW
======================================================================

Lifecycle:
1. Enterprise procures or imports FIDO device inventory
2. Admin registers each FIDO device in WIAM Device Registry
3. Device is assigned to a target user/account before physical delivery
4. Device status becomes ALLOCATED
5. User performs first-time login using Magic Link
6. After successful Magic Link verification, user is allowed to enroll the assigned FIDO device
7. After user completes FIDO enrollment, device status becomes ACTIVE
8. Device can later be revoked, returned, reissued, replaced, retired, or decommissioned

Rules:
- Every device is dedicated to one account/user at a time
- Magic Link is allowed only for first-time onboarding/bootstrap
- After successful FIDO activation, Magic Link must be disabled for normal login

======================================================================
C. ARCHITECTURE RULES
======================================================================

- Keep Auth Core independent
- Device Registry is a foundational platform capability
- Credential enrollment continues in Credential Service
- Device assignment must exist BEFORE enrollment
- FIDO credential enrollment must validate assignment before allowing activation
- Magic Link is a bootstrap auth method only
- Multi-tenant isolation mandatory
- All device actions must be audited
- All magic-link actions must be audited
- Device assignment must integrate with existing users/accounts
- No cross-tenant assignment allowed

Relationship model:

TENANT
  -> USER
  -> ACCOUNT
  -> DEVICE_REGISTRY
  -> DEVICE_ASSIGNMENT
  -> ACCOUNT_BOOTSTRAP_STATE

Device ownership model:
- DEVICE_REGISTRY = enterprise-owned FIDO device inventory
- DEVICE_ASSIGNMENT = mapping of FIDO device to user/account
- Magic Link = temporary bootstrap method to let user enter onboarding
- FIDO credential enrollment = proof that assigned device is activated for auth use
